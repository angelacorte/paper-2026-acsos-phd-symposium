\begin{abstract}
Swarm robotic systems are increasingly relevant for missions in which many autonomous devices must operate in dynamic and potentially safety-critical environments.
%
Programming such systems remains challenging:
developers need to express objectives at the collective level,
while their execution is distributed across devices that may move,
fail, join or leave the system,
without a centralized point of coordination.
%
\acl{ac} addresses this issue by providing macroprogramming abstractions for specifying resilient and self-organizing collective behaviors.
%
However,
current \acl{ac} applications typically execute a single collective program and offer limited support for runtime adaptation,
preemption, concurrent applications, and safety guarantees during transient phases.

This research investigates the foundations of a \acl{cros} built on \acl{ac}:
a middleware layer that lifts ROS-inspired capabilities from individual robots to the collective level.
%
The paper outlines the main requirements for such a system and reports ongoing contributions on morphogenetic formation control,
runtime path replanning, self-stabilizing consensus, collective sensing, and safety guarantees.
%
These results provide initial building blocks towards a collective operating layer for swarm robotics,
with future work focusing on safe runtime adaptation and support for multiple concurrent collective processes.
\end{abstract}

aggregate computing, distributed systems, self-organizing systems, collective operating systems

\section{Overall Problem and Motivation}\label{sec:overall-problem-and-motivation}
Swarm robotics are characterized by the presence of a large number of heterogeneous devices,
which are mobile and resource-constrained.
%
An example is a network of surveillance drones,
which have to coordinate among each other to monitor a certain area,
while maintaining a certain swarm formation, avoiding collisions and respecting no-fly zones
and possibly allowing law enforcement to intervene in case of emergency.
%
Also environmental monitoring systems and search-and-rescue operations can benefit from swarm robotics,
such as wildfire monitoring and disaster response.

Those devices must coordinate among each other to achieve global objectives,
while adapting to changes in the environment and in the network topology,
possibly without a centralized point of coordination,
which can introduce a single point of failure and limit the scalability of the system.
%
Those systems are often deployed in dynamic and unpredictable environments,
where devices may fail or be added/removed at any time.

The design of such systems is challenging,
as it requires to ensure that the system can autonomously adapt to changes in the environment and in the network topology,
while still achieving its goals.

A possible approach to design swarm robotics systems is through macroprogramming~\cite{casadei22},
which allows developers to focus on the desired global behavior of the system,
rather than on the individual behavior of each device.
%
On top of this concept,
is built the \ac{ac} paradigm~\cite{DBLP:journals/computer/BealPV15},
which provides a programming model that allows to express the collective-level behavior of the system through composition of \emph{computational fields}~\cite{JLAMP2019},
a distributed data structure that maps devices to their values,
and that can be manipulated through a set of high-level operators.

By design,
\ac{ac} is resilient to failures and scalable to large numbers of devices,
as it relies on local interactions and \emph{self-organization} to achieve global objectives.
%
Over the years, reusable building blocks~\cite{DBLP:journals/tomacs/ViroliABDP18}
supporting self-stabilizing~\cite{DBLP:journals/cacm/Dijkstra74} \ac{ac} behaviors
have been proposed and later used in high-level libraries such as MacroSwarm~\cite{DBLP:journals/lmcs/AguzziCV25},
which supports pattern formation in swarm robotics.
%%
%Moreover,
%a general formal framework for the specification of collective computational processes has been proposed~\cite{EAAI2020-processes},
%which are temporary distributed computations carried out by dynamic groups of devices,
%as well as a set of algorithms for efficient propagation and termination of such concurrent execution processes~\cite{DBLP:journals/fgcs/AudritoCT24}.

However,
typical \ac{ac} applications run a single although complex algorithm on each device,
without preemption or possibility of changing dynamically the behavior of the system at runtime,
and no users permission definition is provided.
%
In the context of swarm robotics,
this is a limitation,
as multiple applications may need to run concurrently on the same devices,
and potentially third-party users (e.g., law enforcement) may need to stop or change the behavior of the system at runtime in case of emergency.
%
Moreover, %% todo ale
\ac{ac} leverages on the \emph{eventual} consistency of the system to achieve self-stabilization,
which may not be sufficient in safety-critical scenarios,
where a temporary inconsistency may lead to unsafe situations,
such as surveillance drones colliding with each other or entering restricted areas.

\section{Objectives and Contributions}\label{sec:objectives-and-contributions}
% todo Obiettivo -> presentare un sistema operativo collettivo per gli sciami basato su macro-programmazione
% todo Perche? -> paradigma con astrazioni globali, dec, ..

This work aims to address \ac{ac} gaps in the context of swarm robotics,
by proposing a set of mechanisms and algorithms that can be used to build the base of a ROS-like~\cite{ros} \ac{cros} based on \ac{ac},
and by providing a set of reusable self-organizing building blocks that support easy global objectives definition and self-organization.
%
The challenges in designing and implementing such systems are manifold,
as they require to ensure that the system can autonomously adapt to changes in the environment and in the network topology,
while still achieving its goals.

The requirements for such swarm systems include
\begin{inlinelist} %%add motivation
    \item the support for concurrent execution of multiple applications on the same device, parallelizing the execution of different tasks;
    \item preemption and runtime mechanism for dynamic changes in the behavior of the system, to allow third-party users to change the behavior of the system at runtime in case of emergency;
    \item mechanisms to ensure safety in the stabilization phase of the system, to avoid unsafe situations;
    \item heterogeneous devices that act as a collective sensor, to gather distributed information from the environment;
    \item collective-aware team action and formations, to coordinate the movement of the devices in a swarm and to maintain a certain formation;% todo aggiugi formazione spaziali per il movimento coordinato
    \item consensus and communication mechanisms between process in different spatial regions. %todo meccanismi di consenso e communicazione tra processi in regioni spaziali differenti, ad esempio uno spazio di memoria condiviso.
\end{inlinelist}

\section{Methods and Results}\label{sec:methods-and-results}

The research work to cover the gaps mentioned in \Cref{sec:overall-problem-and-motivation} and \Cref{sec:objectives-and-contributions} is currently ongoing,
and it is being carried out through a bottom-up approach,
starting from specific scenarios and use cases that aim to address individual gaps,
and then generalizing the solutions to be applicable to a wider range of scenarios.

Practical \ac{ac} systems are typically supported by \acp{dsl}
implementing variants of the \ac{fc}~\cite{DBLP:journals/tocl/AudritoVDPB19,DBLP:journals/jss/AudritoCDSV24},
and by libraries layered on top of them.
%
Such \acp{dsl} may be implemented either as standalone languages~\cite{DBLP:conf/sac/PianiniVB15}
or as internal languages embedded in mainstream programming environments,
including Scala~\cite{DBLP:journals/softx/CasadeiVAP22},
C++~\cite{DBLP:journals/scp/AudritoT24},
Kotlin~\cite{DBLP:conf/acsos/Cortecchia24},
and Python~\cite{DBLP:journals/corr/abs-2603-29999}.
%
In this work,
we use the Kotlin-based \ac{dsl} \emph{Collektive} as the reference \ac{ac} language,
for its portability to different platforms.

For what it concerns the requirements regarding the formation of spatial structures,
a \emph{morphogenetic} approach to structure formation has been proposed~\cite{DBLP:journals/comintsys/CortecchiaCCP26,DBLP:conf/acsos/CortecchiaPCC24},
which allows to create resilient structures in a self-organizing and self-adaptive way,
based on information about the environment.
%
The base approach is to leverage resource flow management in a network of devices and to distribute them towards the most successful area of the network,
but it can be generalized to specifically create resilient structures that follows mathematically-defined shapes.

About the requirements regarding runtime changes in the behavior of the system,
a runtime path replanning mechanism has been proposed~\cite{DBLP:conf/acsos/AguzziBCMPPV25}.
%
It deals with different devices in an environment that are assigned with a set of tasks to be executed.
%
The devices have to coordinate among each other to achieve the global objective of executing all the tasks,
while adapting their execution plan to changes such as a device failure,
meaning that the tasks assigned to the failed device have to be reassigned to other devices in the network.

Considering the consensus requirements,
a self-stabilizing \emph{min-max gossip} algorithm has been proposed~\cite{DBLP:conf/coordination/CortecchiaPV26},
to provide an adaptive and resilient mechanism to reach consensus on a certain selector among a set of distributed devices,
for example the minimum or maximum value of some sensed information.
%
It differs from classic approaches as it is resilient to failures and dynamic changes in the environment,
without the need of parametrization to reset the gossip memory when the network topology changes.

From the perspective of collective sensing and monitoring,
a set of self-organizing algorithms have been proposed through \ac{dpf}~\cite{DBLP:conf/dcoss/CortecchiaDCCPV26,DBLP:conf/acsos/CortecchiaDCCV26},
an approach for state estimation in distributed systems.
%
\ac{ac} has been used to improve the resilience of the system to failures,
and to provide a better neighboring estimation without the need of exchanging excessive information among devices.

Finally,
to avoid unsafe situation while the system is converging to a stable state,
a safety filter mechanism is ongoing work.
%
By leveraging \ac{cbf}~\cite{DBLP:conf/eucc/AmesCENST19} and \ac{clf}~\cite{DBLP:journals/ieeejas/LiWYWH23} theory applied to \ac{ac} \footnote{\url{https://github.com/angelacorte/carol-experiments}}. %%todo, add reference to the paper when (if) accepted,
%
\ac{cbf} and \ac{clf} are used to define a set of constraints on the system,
such as minimum or maximum distance between devices,
which are then used to enforce safety on the system at runtime,
while computing adaptive swarm-level objectives.
